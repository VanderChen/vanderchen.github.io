Posted on 2021-12-23 Edited on 2021-12-27
Reading note of [Onix. A Distributed Control Platform for Large-scale Production Networks](https://www.usenix.org/legacy/events/osdi10/tech/full_papers/Koponen.pdf)

预备知识—— SDN vs 传统网络

传统网络——固定路由

  • 流量路径的灵活调整能力不足：传统分布式控制算法采用最短路径进行路由计算，而有时候最短路径已经拥塞，系统却没有办法进行自动调节，只能看着最短路径拥塞，而实际有其他路径却空闲着。
  • 传统网络协议复杂，运行维护复杂：数以千计的协议族标准，设备厂家各有自己的扩展，运行维护非常复杂。
  • 传统网络的新业务创新速度太慢：当要在传统网络中部署新业务时，讨论需求、定义标准、达成一致、发布标准、设备供应商支撑标准、对现网设备进行升级，整体过程从需求提出到业务支撑，需要长达3-5年的时间。

SDN——动态路由

转控分离、集中控制、开放接口。

Onix

Onix, a platform on top of which a network control plane can be implemented as a distributed system.

Requirements

  • Generality: 控制平台的 API 允许应用程序提供在各种上下文中的功能
  • Scalability: 适应快速增长的网络
  • Reliability: 能够处理设备失效
  • Simplicity: 需要简化构造管理应用程序的任务
  • Control plane performance: 控制面不应引入重大的额外延迟或以其他方式阻碍管理应用程序。不需要最优时延，而是与通用性做一个trade-off

Components

  • Physical infrastructure

    这包括网络交换机和路由器，以及任何其他网络支持的元素（例如负载平衡器）

  • Connectivity infrastructure

    控制面和Onix之间的通信，分为In-band 和Out-band（control traffic 和data traffic分离）

  • Onix

    Onix 是一个分布式系统，运行在一个或多个物理服务器的集群，每个它可以运行多个 Onix 实例。作为控制平台，Onix 负责给予控制逻辑对网络的编程访问

  • Control logic

是Onix的核心概念，可以理解为最小的控制单元，后文的拓展性和可靠性都依赖于NIB

Partition

当物理节点的过多，超出一个server的memory size，则分区。分区后使用DHT存在多个NIB中。

Aggregate

当流表变化过于频繁，超出了CPU的限制，则聚合。将一组拓扑视为一个逻辑上的node。

Partition further

没太看懂，大意是说虽然拓扑被存放在了多个NIB中，但是仍会做一个联合决策。具体来说，NIB中路由自治，controller只控制NIB间路由。

Distributing the NIB

State Distribution Between Onix Instances

简单来说，Onix实现了一个事务持久性数据库（transactional persistent database）。由用于传播的复制状态机支持所有需要持久性和简化的状态更新一致性管理。要将复制的数据库与 NIB 集成，Onix 有用于数据库交互的导入/导出模块。这些组件加载和存储实体来自/到事务的声明及其属性数据库。对于需要高更新率和可用性的网络状态能力，Onix 提供了一个单跳的、最终一致的、memory-only DHT。

Network Element State Management

Onix 设计并未规定特定的协议管理网元转发状态。而是，应用程序的主要接口是 NIB，任何网络中的元素支持的合适的协议可以在幕后使用以将 NIB 实体保持在与实际网络状态同步
